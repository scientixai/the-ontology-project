1.TL;DR one container, all the categories operators actually deal with

Event is the single container for clinical-trial occurrences requiring tracking, reporting, or escalation. Per first-principles.md universal-foundation posture, one Event entity covers Adverse Event, Serious Adverse Event, Protocol Deviation, Data Discrepancy, Safety Signal, Safety Report, and Other Clinical Event — discriminated by eventCategory enum, not by entity type. Per ICH E2A / E2B(R3) / 21 CFR 312.32, regulatory reportability is a first-class attribute with submission-timeline tracking. Per ICH E6(R3) Section 6, deviations carry category and impact. PROV: Event is prov:Activity (the event IS the thing that happened); reportedBy → Person carries prov:wasAssociatedWith; derivedFrom → VisitObservation carries prov:wasInformedBy (the operator-meaningful escalation chain). Un-flag-misses three relationships: Participant.hasAdverseEvent, Visit.hasAdverseEvent, VisitObservation.escalatedTo. Closes the OOUX-locked-9 top-level set.

2.Architectural anchor

┌─→ Study (forStudy — every Event ties to a Study) │ ├─→ Participant (forParticipant — most events) │ Event ────────────────────┼─→ StudySite (forStudySite — locality of occurrence) │ ├─→ Visit (duringVisit — when detected at a Visit) │ ├─→ VisitObservation (derivedFrom — escalation chain · prov:wasInformedBy) │ ├─→ InvestigationalProduct / Lot / Kit (causality chain) │ ├─→ Person (reportedBy / assignedTo · prov:wasAssociatedWith) │ ├─→ Event (partOf · causedBy[*] — aggregation + causal lineage) │ ├─→ OversightBody (reviewedBy[*] — DSMB / IDMC / IRB review) │ └─→ Document (documentedIn[*] — MedWatch, CIOMS, CAPA)

Event reaches every other top-level. The forward-references that pre-date this lift are Participant.hasAdverseEvent → Event, Visit.hasAdverseEvent → Event, and VisitObservation.escalatedTo → Event — all flagged-missing in earlier lifts and resolved here. The derivedFrom relationship is the operator-meaningful escalation chain: a routine vital-sign observation that flags as elevated escalates into a tracked Event with explicit lineage.

4.Architectural decisions

Decision 1 — Single Event entity with category discriminator (universal foundation)

Per FIRST-PRINCIPLES universal-foundation posture: don't split into separate AdverseEvent / Deviation / Discrepancy / SafetySignal classes. One TOP entity shape; eventCategory enum carries the operator-grounded discrimination. Same architectural moat as Activity + Task (v0.5), OversightBody (v0.5.1), and InvestigationalProduct (v0.6.0). Standards-up vendors model N entity types per category; TOP carries one universal shape and projects to FHIR / SDTM / E2B at the adapter layer.

Decision 2 — Universal-grade attributes coexist with clinical-research-specific attributes for v0.7

The migration table targets topc:Event (universal primitive shape — every regulated workflow has Events) plus topcr:Event subClassOf topc:Event (clinical-research specialization adding causality / MedDRA / CTCAE / IP-causality / protocol-deviation discipline). For v0.7 those concerns co-locate under the existing top: prefix; the upcoming namespace-refactor PR performs the split with no shape changes — it is purely a prefix renaming + subClassOf addition.

Decision 3 — Event is prov:Activity

The Event IS the thing that happened. Visit was prov:Activity for the same reason — the visit IS the activity. The recorded fact (the Event entity itself) is both the prov:Activity and its audit-trail entry. eventDate carries prov:atTime; reportedBy carries prov:wasAssociatedWith; derivedFrom carries prov:wasInformedBy (the escalation chain from VisitObservation). The provenance chain is queryable by every PROV-aware tool downstream.

Decision 4 — Severity uses one enum spanning both AE/SAE and non-clinical scales

AE/SAE per ICH E2A grade severity as MILD / MODERATE / SEVERE / LIFE_THREATENING / FATAL. Non-clinical events (DEVIATION, DISCREPANCY) typically use LOW / MEDIUM / HIGH / CRITICAL. Two scales kept in one enum for foundation simplicity; the projection adapter restricts to the right scale per category. CTCAE grading is a separate field (ctcaeGrade) because CTCAE 5.0 grading is more granular and only applies to AE/SAE.

Decision 5 — hasAdverseEvent kept as the operator-vocabulary forward reference; broader category access via Event.forParticipant

Operators say "adverse events" out loud during their workday. Participant.hasAdverseEvent and Visit.hasAdverseEvent are kept as the named forward references — they're the operator-vocabulary view filtered to AE/SAE category. Broader category access ("all events for this participant", "all deviations during this visit") is reachable via Event.forParticipant / Event.duringVisit inverse traversal. Don't create both hasAdverseEvent and hasEvent; that's redundant.

7.Worked example Maria's Grade 3 immune-related hepatitis SAE

Day 13 post-C1D1: Grade 3 immune-related hepatitis SAE
SCENARIO · SAE

Maria Rodriguez (Participant onco-423-mskcc-p001) received Drug X (anti-PD-L1 monoclonal antibody, BIOLOGIC, blinded) at her C1D1 visit on 2026-04-12. On 2026-04-25 (Day 13 post-dose), routine follow-up labs returned ALT 8× ULN, AST 6× ULN, total bilirubin 2.1 mg/dL — asymptomatic but Grade 3 by CTCAE 5.0. Investigator Patel reviews, classifies as immune-mediated hepatitis (consistent with PD-L1 mechanism per IB Section 6.4), interrupts Drug X dosing, and reports the SAE the same day.

Event { eventId: ":Event:onco-423-mskcc-p001-sae-001", eventCategory: "SERIOUS_ADVERSE_EVENT", ctcaeGrade: "GRADE_3", causality: "PROBABLY", expectedness: "EXPECTED", regulatoryReportable: true, regulatoryReportDeadline: 2026-05-10T23:59:59Z, regulatoryReportSubmitted: true, regulatoryReportSubmittedDate: 2026-05-08T11:22:00Z, meddraPreferredTerm: "Immune-mediated hepatitis", actionTakenWithIp: "DOSE_INTERRUPTED", outcome: "RECOVERING", forParticipant: → Maria, relatedToInvestigationalProduct: → Drug X, relatedToLot: → DRX-2026-0117, relatedToKit: → KIT-00042, reportedBy: → Patel, reviewedBy: → ONCO-423 DSMB }

Validates against all 5 v0.7.0 invariants: SAE has regulatoryReportable=true (#41); ctcaeGrade is set on AE/SAE (#42); protocolDeviationCategory is unset (#43 satisfied vacuously); regulatoryReportSubmitted=true paired with submittedDate (#44); deadline is satisfied (#45 not triggered — submitted before deadline).

Out-of-window labs as DEVIATION
SCENARIO · DEVIATION

Day 8 safety labs were due 2026-04-20. MSKCC outpatient labs closed for severe weather; labs drawn the following day (Day 9). Outside protocol-specified ±0 day window for Day 8 but within ±1 day operational tolerance. Coordinator Jones documents as DEVIATION with category OUT_OF_WINDOW_VISIT, impact MINOR, regulatoryReportable=false, status CLOSED. Demonstrates the deviation-discipline path through the same Event entity — same shape, different category, different relevant attributes.

8.CTAs operator-meaningful actions

  • Report Adverse Event — Investigator / Coordinator. Create Event with eventCategory=ADVERSE_EVENT (or SAE), populate severity / eventDate / MedDRA term / CTCAE grade / causality / actionTakenWithIp / outcome.
  • Escalate to SAE — Investigator / Safety. Reclassify AE → SERIOUS_ADVERSE_EVENT, set regulatoryReportable=true, populate regulatoryReportDeadline per E2B timeline, link parent AE via partOf.
  • Document Protocol Deviation — Coordinator / Monitor. eventCategory=DEVIATION, populate protocolDeviationCategory + protocolDeviationImpact, link the affected Visit via duringVisit.
  • Submit Regulatory Report — Sponsor Safety / Regulatory. Set regulatoryReportSubmitted=true with regulatoryReportSubmittedDate; attach E2B XML or CIOMS PDF via documentedIn.
  • Aggregate Safety Signal — Sponsor Safety / Pharmacovigilance. eventCategory=SAFETY_SIGNAL, link constituent AE Events via causedBy[*], attach DSMB review via reviewedBy.
  • Close Event — Investigator / Safety. Transition status to CLOSED after final outcome and required regulatory reporting. NGSI-LD temporal history preserves the lifecycle.

9.Forward references resolved

  • Participant.hasAdverseEvent → Event — was flagged-missing since v0.4 (Participant lift). Now resolves to the Event entity. Operator vocabulary retained: the relationship label remains hasAdverseEvent (not hasEvent); Event is reachable in full-category breadth via Event.forParticipant inverse.
  • Visit.hasAdverseEvent → Event — was flagged-missing since v0.5 (Visit lift). Resolves; broader category access via Event.duringVisit inverse.
  • VisitObservation.escalatedTo → Event — was flagged-missing since v0.5. Resolves; carries prov:wasInformedBy semantics. The escalation chain from a routine observation to a tracked Event is now first-class queryable provenance.
