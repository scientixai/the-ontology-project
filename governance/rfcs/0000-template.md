## Motivation

What problem does this RFC solve? Who is affected? What concrete operator workflow, downstream consumer, or contributor pain point surfaced this? Keep this tight — the reader needs to understand the *why* before judging the *what*.

If a specific moment forced the question (an operator's pushback, a TermBoard import failure, a cross-WG conflict), quote it. The pivotal moments are usually the strongest motivation.

## Proposal

The concrete change being proposed. Names of artifacts. New URIs. Specific class / property / file additions or modifications. Diagrams welcome.

If the proposal includes code or Turtle, show it inline so reviewers can read the substance without context-switching.

## Alternatives considered

What other options were on the table, and why are they not the recommendation? At least one alternative is almost always worth describing — the "do nothing" alternative is valid and often instructive.

A useful framing for each alternative:
- **What it would change**
- **What it preserves**
- **Why it was not chosen**

## Open questions

Anything the author is uncertain about and wants reviewer guidance on. Better to surface ambiguity here than to merge an RFC that hides it.

If the RFC is asking for a decision between two paths, list both paths and ask explicitly.

## Consequences

What ratification of this RFC commits the project to. Be honest about cost as well as benefit:

- **What gets easier**
- **What gets harder**
- **What downstream consumers must adapt to**
- **What follow-on work this RFC creates**
- **What this RFC forecloses** (decisions that become harder to revisit later)
