Why this view is impossible today. A data manager asking “show me the complete story of this data point” must open five systems: the protocol PDF (for the item requirement), CTMS (for the delegation log proving the CRC was authorised), the EDC system (for the value and audit trail), the monitoring report (for the SDV record), and the query management system (for the query chain). Cross-referencing these takes hours per data point — multiplied across thousands of items and 149 participants.

What TOP does differently. Every stage in this chain is a first-class entity in the TOP graph. The protocol item requirement (cr:CRFItem) is linked to the observation via cr:instantiatesTemplate. The SDV record is a cr:SDVRecord linked to the observation via cr:sdvRecord. The query is a cr:Discrepancy linked via cr:openQuery. The delegation is on the hcls:Person who performed it. A single join=inline&joinLevel=4 call returns the whole chain in one response.

The investement thesis. The “innovation tax” is real: building this view from point-solution APIs requires weeks of integration work per sponsor. TOP makes the graph the integration layer. Point solutions (EDC, LIMS, eTMF, LMS) remain; the graph is the connective tissue. Queries that took weeks become milliseconds.

  • Observation must reference the (pseudonymous) StudySubject.
  • Observation must reference the StudyEvent it was captured at.

  • DataElementDef should carry a PHI classification (NotPHI/LimitedPHI/FullPHI/Restricted).
cr:QueryShapecr:QueryViolation
  • Query must be anchored to a target entity (cr:queryOn).
  • Query must carry a resolution status (open/answered/closed).
