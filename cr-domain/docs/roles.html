Roles, phases & actions

Who can do what, when — the operator-grounding layer. Authorization binds directly to OWL agent type classes; no separate Role intermediary. A real hcls:Person is typed as one of these classes; their contextual title within a Study is a plain string (hcls:actsAs).

Design principles

  • Agent types are OWL classes, not named individuals. A Person typed as cr:Investigator inherits all authorizations declared on that class.
  • Delegation at instance time: the PI delegates to staff via cr:Delegation acts; top:authorizedBy on the Person records who delegated to them. The DoA log is a SPARQL projection of those acts, not a maintained spreadsheet.
  • Credentials at instance time: top:hasCredential on Person → top:Credential. A delegated task requires both delegation and the matching credential before it may proceed.
  • Regulatory constraint is additive: some actions carry a top:RegulatoryLaw individual (e.g. ICH E6(R3) mandate to delegate at the site level). Most do not.

Authorization query pattern

To answer "What am I authorized to do right now?" given an agent type and phase:

SELECT ?action ?label WHERE {
  ?action hcls:authorizedAgentType cr:SiteCoordinator ;
          hcls:occursIn ?phase ;
          rdfs:label ?label .
  # include sub-phases:
  cr:ConductPhase hcls:contains+ ?phase .
}
