Schedule of Activities

The visit schedule most systems flatten into a 2D table with the timing buried in footnotes — modeled as a timeline: a planned template (windows, relative timing, ordering, repeats), projected onto a participant once the anchor is known, then reconciled against what actually happened.

The bitemporal payoff a static SoA can’t give: the planned schedule is reconstructable as-of any protocol amendment, and a missed (planned-but-unrealized) visit is a detective query. Out-of-window is a Warning (deviation candidate); broken hard ordering is a Violation.

Operator screen — Schedule of Activities (OPP-101)

The impossible screen. A CRC at Denver Cancer Center opens the SoA view for S-042. They see planned dates with protocol-resolved visit windows, actuals from EDC, out-of-window flags computed automatically, and the system knows which protocol version governed each visit. Today this requires exporting from CTMS, EDC, and the protocol PDF and cross-referencing by hand. With TOP: one broker query, sub-200 ms.

Why this view is impossible today. A CRC assembles this by hand: open the protocol PDF for windows, query CTMS for actuals, subtract dates, flag deviations. For 149 participants across 30 sites that is weeks of reconciliation work.

What TOP does differently. The cr:ScheduleTimeline carries the planned template — windows, ordering, repeats. When a participant’s anchor date (enrollment) is known, cr:PlannedVisit entities are projected per participant with resolved target dates and windows. The windowStatus flag is written at ingestion time by the transformer when the actual visit date arrives — so this query is a lookup, not a computation.

Bitemporality payoff. When OPP-101 issues Amendment 1, a new cr:ScheduleTimeline with a later validFrom is ingested. Participants enrolled before the amendment remain governed by v1.0; new enrollees get amended windows. Which version governed which visit is a query — not an audit trail dig.

Data sources unified in this view. CTMS (actual visit dates), EDC (activity completion status), Protocol design system (windows, ordering). Three separate point solutions; one broker call.

cr:ParticipantScheduleParticipant ScheduleEvidenceThe schedule projected for a specific subject from a ScheduleTimeline once an anchor date (e.g. enrollment) is known. Bitemporal: re-projected when the governing protocol version changes.
cr:PlannedActivityPlanned ActivityConstraintA planned activity within an encounter (USDM ScheduledActivityInstance) — e.g. ECG, blood draw, PRO. Positioned by a cr:Timing and/or an ordering edge (cr:precedes).
cr:PlannedEncounterPlanned EncounterConstraintA planned visit slot on the timeline (USDM Encounter). Positioned by a cr:Timing relative to an anchor; actual cr:Visits are reconciled against it.
cr:PlannedVisitPlanned VisitConstraintA projected planned visit for a subject: a target date and a resolved window (start/end dates), against which the actual visit is reconciled.
cr:RepeatingPlannedActivityRepeating Planned ActivityConstraintA planned activity that recurs on an interval until a stop condition (e.g. a daily participant diary until Day N; dosing every 7 days).
cr:ScheduleTimelineSchedule TimelineEvidenceThe planned Schedule of Activities for a protocol version, as a timeline of planned encounters and activities. Bitemporal: a protocol amendment supersedes it, so the schedule is reconstructable as-of any date.
cr:TimingTimingConstraintA relative-timing rule: an offset (days or hours) from an anchor — another node, or an anchor event such as enrollment — with an allowed window. The footnote logic, made explicit.
cr:WindowWindowConstraintAn allowed tolerance around a planned time: a lower and an upper offset in days (e.g. [-2 days, +2 days]).
