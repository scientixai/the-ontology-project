Implementation guide — build your own graph

This is a reference asset, not a runtime: standard W3C artifacts (RDF / OWL / SHACL / SPARQL / SKOS / PROV) you load into any graph store to give your own clinical-research knowledge graph the operator model, the constraints, and the standard views — out of the box. Nothing here is vendor-specific.

A deployable reference architecture and demo (an AWS Garnet / NGSI-LD CDK deployment plus a mapping workbench) ship separately with the community release. This page is about leveraging the reference on your own stack — today, with tools you already have.

2 · Get the artifacts

Two equivalent sources. The repository (cr-domain/) holds the source modules — clone or vendor them. The prebuilt bundles in dist/ are a single merged file per layer, regenerated deterministically by docs/build_dist.py and pinned by checksum:

sha256sum -c SHA256SUMS        # verify the bundle you pinned, in dist/

The download links above are served from this site's dist/ folder (and mirrored in the repo). When self-hosting, serve dist/ alongside these pages, or reference the repository directly — the SHA256SUMS lets consumers pin an exact version regardless of where they fetched it.

Pick a serialization. Every bundle ships in three byte-equivalent forms: .ttl (Turtle — read, author, load), .jsonld (JSON-LD — web/JS toolchains; the separate *.ngsi-context.jsonld is the broker term map), and .nt (N-Triples — one canonical triple per line: stream-load into large stores, line-diff cleanly across versions, and the form the SHA256SUMS hashes over). Same triples, different envelope — choose by tool, not by content.

3 · Load it into your store

It's standard RDF, so any triplestore or RDF-capable graph DB works. A few concrete starts:

Python (RDFLib + pySHACL)

from rdflib import Graph
g = Graph().parse("dist/top-cr-v1.ttl", format="turtle")   # the merged model
g.parse("my-study-data.ttl", format="turtle")              # your instance data

Neo4j (neosemantics / n10s)

CALL n10s.graphconfig.init({handleVocabUris:'SHORTEN'});
CALL n10s.rdf.import.fetch('https://<host>/dist/top-cr-v1.ttl','Turtle');
CALL n10s.rdf.import.fetch('https://<host>/dist/top-cr-shapes-v1.ttl','Turtle');

Apache Jena / Fuseki, GraphDB, Blazegraph, …

# load the Turtle bundle into a dataset, then query with SPARQL 1.1
fuseki> :data UPLOAD dist/top-cr-v1.ttl
# or RDF4J / GraphDB: import top-cr-v1.ttl into a repository

NGSI-LD broker (e.g. Scorpio)

The broker is the runtime serving layer — not another triplestore for the model. You load instance entities into it (after reasoning and validating offline); see §6 — Serve it below for the @context and the term mapping.

4 · Validate with the shapes

Run the graded SHACL shapes against your data — hard structural rules surface as Violation, risk-proportionate ones as Warning:

from pyshacl import validate
conforms, _, report = validate(
    my_data_graph,
    shacl_graph="dist/top-cr-shapes-v1.ttl",
    ont_graph="dist/top-cr-v1.ttl",
    inference="rdfs", advanced=True)   # advanced=True enables the SPARQL constraints

Any SHACL engine works (pySHACL, Apache Jena shacl, TopBraid). The test harness tests/run_tests.py is a worked example of validating instance data. The shapes are vended in all three serializations (top-cr-shapes-v1.{ttl,nt,jsonld}) and pinned by SHA256SUMS — same graded rules, whichever your engine prefers.

5 · Reason over it — bring your own reasoner

This reference ships the ingredients, not the baked cake: it does not run a reasoner. Classification, materialization, and entailment are the consumer's job — which reasoner, at what profile, and when you materialize are your deployment decisions, not the reference's. What the reference guarantees is that the ingredients are sound: the OWL is authored to the OWL 2 DL typing discipline, and a CI gate (tools/owl_lint.py) proves it on every change — no illegal punning, every term declared, every domain/range well-typed — so a reasoner you point at it returns sound entailments, not garbage.

Profile. The model is deliberately light: one rooted class hierarchy, object/datatype properties with rdfs:domain/rdfs:range, and a few owl:inverseOf pairs. That sits inside both OWL 2 RL and OWL 2 DL, so it runs unchanged in rule reasoners (owlrl, RDFox, GraphDB rules) and in tableau/EL reasoners (HermiT, ELK, Pellet). Layer your own restrictions, equivalences, or property chains on top as your use demands — the reference stays the shared base.

What reasoning gives you over the reference + your data:

  • Subclass entailment — a cr:DoseLimitingToxicity is entailed a cr:AdverseEvent → top:Outcome → top:Core, so it carries the Universal DNA.
  • Domain/range typing — asserting ex:s cr:sponsoredBy ex:o types ex:s a cr:Study and ex:o a cr:Sponsor.
  • Inverse fills — cr:hasQuery↔cr:queryOn, cr:specimen↔cr:hasCustodyEvent, and the other inverse pairs.
  • Whatever you add — defined classes (necessary&sufficient), property chains, or owl:disjointWith (whose violation a DL reasoner reports as an inconsistency).

Python — OWL 2 RL, pure-Python (no Java)

from rdflib import Graph
from owlrl import DeductiveClosure, OWLRL_Semantics
g = Graph().parse("dist/top-cr-v1.ttl", format="turtle")
g.parse("my-study-data.ttl", format="turtle")
DeductiveClosure(OWLRL_Semantics).expand(g)   # forward-chain; g now holds the entailed triples

Python — OWL 2 DL via HermiT (owlready2, needs Java)

from owlready2 import get_ontology, sync_reasoner
onto = get_ontology("dist/top-cr-v1.ttl").load()
onto.load("my-study-data.ttl")
with onto: sync_reasoner()                     # HermiT: classify + detect inconsistency

Java / CLI — ELK or HermiT

robot reason --reasoner ELK --input top-cr-v1.ttl --output inferred.ttl
# or drive HermiT / ELK directly through the OWL API

Reason offline, then serve. Forward-chain once, persist the entailed triples, and load those into your runtime store or broker — the broker doesn't reason (next section). Re-materialize when the model or your data changes.

6 · Serve it — NGSI-LD as the runtime

The runtime is an NGSI-LD context broker (e.g. Scorpio on AWS Garnet) — not a reasoner, not a validator. It stores entities and serves them: relationships, temporal/“as-of” queries, subscriptions, federation, and single-pull retrieval. It does not load OWL axioms, classify, or run SHACL.

So the division of labor is clean: reason (§5) and validate (§4) offline; load the resulting asserted + entailed + validated entities into the broker; the broker serves them fast at the glass.

# load entities under the published @context so JSON terms resolve to cr:/top: IRIs
Link: <https://<host>/dist/top-cr-v1.ngsi-context.jsonld>; rel="http://www.w3.org/ns/json-ld#context"
# top:observedAt -> NGSI-LD core observedAt (valid time)
# top:recordedAt / top:supersededAt -> createdAt / modifiedAt

Single-pull retrieval. The model is shaped so each operator screen resolves to one self-contained entity that already carries its green-check facts — no recursive lookups at the glass. tools/ngsild_view.py builds these views, and the per-flow pages show the screen each one feeds.

A deployable Garnet / NGSI-LD CDK stack and a mapping workbench ship with the community release; this section is the architecture, not that deployment.

7 · Project to standards (define once, view many)

The standards are queries, not separate databases. Run a projection over your graph to emit that view — e.g. projections/sdtm_dm.rq (CDISC SDTM demographics), projections/usdm_study.rq, projections/soa_table.rq (the SoA table as a view of the timeline), or projections/planned_vs_actual.rq (visit reconciliation). Same native graph, many standardized outputs, guaranteed consistent.

8 · Ingest an M11 / USDM protocol

To turn a protocol into a TOP-grounded graph: load the vendored ontology/vendor/usdm/usdm-v4.ttl (the USDM class model) and apply crosswalks/usdm-to-cr.ttl to map USDM constructs onto cr-core — respecting the plan↔execution boundary (USDM is design-time; cr-core is execution + provenance). tools/usdm-rdf-gen/ingest_usdm.py does exactly this for real on the public, MIT-licensed CDISC Pilot (LZZT) study — emitting examples/usdm-cdisc-pilot-ingested.ttl, which validates green — and examples/usdm-cdisc-pilot-conformant.ttl is a clean hand-authored illustration of the target shape. USDM classes and codelists are NCIt-anchored (verified against the NCI Thesaurus), so your graph inherits a verifiable terminology spine.

9 · Honor the conventions (so graphs line up)

  • Universal DNA on every entity: top:identifier + top:observedAt (valid time) + top:status. This is what makes any two TOP graphs align by construction.
  • Bitemporal envelope: add top:recordedAt (and top:supersededAt on correction) so you can reconstruct any past state “as of” a date. Corrections are new assertions, never rewrites.
  • Provenance: prov:wasAttributedTo on every fact — who asserted it.
  • NGSI-LD mapping: top:observedAt ≈ NGSI-LD observedAt; top:recordedAt/supersededAt ≈ createdAt/modifiedAt.

See the Foundation page for the model and the conventions.md file for the full envelope.
