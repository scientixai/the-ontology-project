Ingestion worked example — LY900018 (USDM v4 → NGSI-LD)

Eli Lilly LY900018 is a nasal glucagon rescue treatment for hypoglycaemia in T1DM / T2DM patients. Its USDM v4 JSON protocol is the reference fixture for the TOP ingestion pipeline. The transformer (examples/usdm/to-ngsi-ld.py) reads the raw USDM document and emits 102 NGSI-LD entities across 10 type buckets, each carrying the Universal DNA and ready to POST to a Scorpio broker.

What USDM gives you

USDM v4 (CDISC Unified Study Definitions Model) is a structured JSON export of a protocol: study versions, arms, epochs, encounters (visits), eligibility criteria, objectives, endpoints, and governance history. It is the ICH M11 machine-readable equivalent — the canonical protocol source.

The transformer does three things:

  1. Structural mapping — USDM StudyVersion → cr:ProtocolVersion; USDM Encounter → cr:Visit; USDM EligibilityCriterion → cr:EligibilityCriterion; etc.
  2. Bitemporality injection — extracts GovernanceDate from the USDM StudyVersion and uses it as top:validFrom on the emitted cr:ProtocolVersion. When the protocol is amended, a new ProtocolVersion entity with a later validFrom is POSTed; the broker keeps both, and a SPARQL query can reconstruct which version was in force on any given date.
  3. URN assignment — every entity gets a stable urn:ngsi-ld:top-cr:{Type}:{localId} built from the USDM object identifier, so the same study ingested twice produces the same URNs and the broker can upsert idempotently.

Entity walk-through

1 Study — the anchor node

One entity per study. All other entities hang off it via Relationship nodes. The sponsoredBy Relationship uses @type: @id in the @context so the broker serialises it as a URI reference — enabling traversal in a single NGSI-LD request with the options=keyValues&attrs=sponsoredBy query parameter.

{
  "id": "urn:ngsi-ld:top-cr:Study:LY900018",
  "type": "cr:Study",
  "@context": [
    "https://uri.etsi.org/ngsi-ld/v1/ngsi-ld-core-context-v1.8.jsonld",
    "https://top.scientix.ai/cr/v1.ngsi-context.jsonld"
  ],
  "identifier":  { "type": "Property",      "value": "LY900018" },
  "status":      { "type": "Property",      "value": "active" },
  "observedAt":  { "type": "Property",      "value": "2026-06-30T00:00:00Z" },
  "sponsoredBy": { "type": "Relationship",  "object": "urn:ngsi-ld:top-cr:Sponsor:EliLilly" }

2 ProtocolVersion — bitemporal protocol snapshot

cr:ProtocolVersion is a top:Versioned entity. The validFrom property records when this version of the protocol was authorised in the world (extracted from the USDM GovernanceDate), independently of when the system ingested it (observedAt). Querying the broker with q=validFrom<=2018-03-01 returns the protocol version that was in force on that date — essential for reconstructing what rules governed a specific visit.

4 Visit — relationship chain across the SoA

cr:Visit entities form a linked list via nextVisit Relationships — enabling a broker to traverse the schedule of activities in sequence without a SPARQL join. Running the transformer yourself

# from repo root
python3 examples/usdm/to-ngsi-ld.py \
    examples/usdm/ly900018-usdm-v4.json \
    --out examples/usdm/out/

# POST to a local Scorpio broker
for f in examples/usdm/out/*.json; do
  curl -s -X POST http://localhost:9090/ngsi-ld/v1/entities \
    -H 'Content-Type: application/ld+json' \
    -d @$f
done

The transformer is idempotent: POST the same file twice and the broker upserts (no duplicate entities). Entities that differ only in observedAt are treated as updates; entities that differ in validFrom or other bitemporal properties create a new version snapshot.

What’s next

  • EDC streaming — a Kafka consumer subscribing to a Castor EDC topic ingests subject-level cr:Assessment and cr:Enrollment entities as they are CRF-captured, linking them to the cr:StudySubject anchor. The PII bridge (cr:Enrollment → hcls:Person) stays encrypted at rest and is not written to the broker.
  • Veeva DDA catch-up — a bulk loader reads an ODM-XML export from S3, resolves subjects and visits against the already-loaded protocol graph, and emits clinical hcls:Observation entities timestamped with the original collection date (validFrom) vs the import date (observedAt).
  • LLM field mapping — sponsor-specific X_WHATEVER fields in Medidata are emitted as cr:OperationalRecord entities with a cx:mappingConfidence score from the LLM inference step. A human reviewer promotes confirmed mappings to canonical cx:Mapping records.
